# Contributing

## Principles

- Keep documents practical, concise, and executable.
- Prefer patterns backed by real operator experience over abstract theory.
- When proposing a new template, show the use case and failure mode it addresses.

## Suggested contribution flow

1. Identify the harness problem clearly.
2. Add or update the smallest useful document.
3. Validate with `scripts/validate.sh`.
4. Keep examples short and reusable.

## Pull request guidance

A good PR should answer:
- What operator problem does this solve?
- What workflow becomes easier after this change?
- What failure or ambiguity does it reduce?

## Style

- Use short sections.
- Prefer bullet lists over long walls of text.
- State tradeoffs directly.
- Avoid hype language.
